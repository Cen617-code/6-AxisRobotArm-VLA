# 6 轴机械臂 VLA 接入 15 天学习与实施计划

## 一、项目目标

目标是在 15 天内，按下面这条主线，把一个轻量级 VLA 能力稳妥接到当前 6 轴机械臂仿真项目中：

`环境测试 -> 单张图像推理测试 -> 输出动作映射 -> 闭环控制`

这份计划的核心原则如下：

- 不跳步，每天只攻克一个主问题。
- 不一开始就追求“大而全”，先打通最小可运行链路。
- 先做“到达目标”，不做抓取。
- Python 负责视觉与 VLA，C++ 负责运动规划与执行。

## 二、当前项目基线

当前仓库已经具备以下基础能力：

- `ROS 2 Jazzy`
- `Gazebo Harmonic`
- `MoveIt2`
- `ros2_control + joint_trajectory_controller`
- C++ 正运动学 / 逆运动学
- 规划避障

当前缺少的关键模块如下：

- Python 侧 VLA 运行环境
- 图像输入链路
- VLA 输出到 MoveIt 执行桥
- 面向闭环实验的统一任务接口

## 三、总体技术分层

为了后续结构清晰，默认采用下面这套分层：

- Python 层负责：
  - 图像读取
  - 文本任务理解
  - VLA 推理
  - 发布 `/vla/action_delta`
- C++ 层负责：
  - 获取当前末端位姿
  - 安全裁剪目标位姿
  - MoveIt 规划与执行
  - 执行结果回传

第一阶段只做“让末端接近目标”，不涉及夹爪控制，不涉及抓取状态机。

## 四、15 天实施计划

### Day 1：确认仿真基线完全可复现

**今日目标**

确认现有 ROS 2 机械臂仿真链路能稳定复现，作为后续 VLA 接入的地基。

**核心任务**

- 重新编译工作区。
- 启动 Gazebo 仿真。
- 启动 MoveIt RViz。
- 手动拖动目标，验证 `Plan & Execute`。
- 运行 `add_obstacles`，确认避障仍然有效。

**当日验收标准**

- 机械臂能在仿真中正常规划与执行。
- 添加障碍物后，规划结果会绕障而不是穿过障碍物。

**当日产出**

- 一份基线检查清单。
- 一组可复用启动命令。
- 关键 topic 列表。

### Day 2：准备独立 Python VLA 环境

**今日目标**

为 VLA 单独准备 Python 运行环境，避免污染现有 ROS 2 工作区。

**核心任务**

- 规划 `vla/` 目录结构。
- 建立独立虚拟环境。
- 确认 `rclpy`、图像库、模型依赖可以共存。
- 检查 GPU 是否可见，至少保底可跑 CPU。

**当日验收标准**

- 能在独立环境中导入 `rclpy`。
- 能导入深度学习框架。
- 能完成 GPU 检测，或明确 CPU 降级可用。

**当日产出**

- 环境依赖清单。
- 安装顺序说明。
- 环境自检脚本列表。

### Day 3：完成最小图像输入链路

**今日目标**

完成单张图像输入与预处理，但先不接 ROS。

**核心任务**

- 准备 1 到 3 张测试图片。
- 完成单图读取、resize、normalize。
- 保存一份可视化结果，确认预处理没有把图像弄坏。
- 定义统一输入接口。

**当日验收标准**

- 脚本可以稳定读取一张图。
- 能输出标准化后的张量或中间结果。
- 输入输出尺寸和数据类型清楚可复现。

**当日产出**

- 离线图像预处理脚本接口说明。
- 测试图片目录规范。

### Day 4：完成第一次离线 VLA 推理

**今日目标**

接入轻量 VLA 后端，完成第一次“图片 + 文本 -> 动作”的离线推理。

**核心任务**

- 优先采用 Octo 作为烟雾测试后端。
- 加载模型。
- 输入一张测试图片和一句任务文本。
- 得到一组可解释的动作值。

**当日验收标准**

- 模型能完成一次完整推理。
- 不出现 OOM。
- 输出为有限数值，不是 `NaN` 或空结果。

**当日产出**

- 第一版 `infer(image, instruction, robot_state) -> ActionChunk` 规范。
- 一份离线推理结果样例。

### Day 5：统一动作输出格式

**今日目标**

把 VLA 原始输出统一成项目内部动作格式，方便后续 ROS 与 MoveIt 接入。

**核心任务**

- 定义 `ActionChunk`。
- 固定字段为：
  - `delta_xyz[3]`
  - `delta_rpy[3]`
  - `confidence`
  - `terminate`
- 第一版先只使用 `delta_xyz`。

**当日验收标准**

- 离线推理结果能稳定转成统一结构。
- 结果可以保存为 JSON。
- 每个字段的含义和单位明确。

**当日产出**

- 动作结构定义文档。
- 示例 JSON 输出。

### Day 6：把 Python 推理节点接入 ROS 2

**今日目标**

把 Python 推理节点作为 ROS 2 节点运行起来，但暂时不驱动机械臂。

**核心任务**

- 接收文本任务，或读取固定任务文本。
- 读取离线图片或缓存图片。
- 发布 `/vla/action_delta`。

**当日验收标准**

- 使用 ROS 2 命令能观察到 `/vla/action_delta` 正常发布。
- 节点启停稳定。
- 发布频率和消息格式清晰可控。

**当日产出**

- VLA ROS 2 节点接口设计。
- 节点启动方式说明。

### Day 7：动作映射到安全目标位姿

**今日目标**

把动作增量安全地映射成机械臂末端目标位姿。

**核心任务**

- 读取当前 `/end_effector_pose`。
- 将 `delta_xyz` 累加成绝对目标位姿。
- 加入单步位移裁剪。
- 加入工作空间边界限制。
- 姿态默认保持当前姿态不变。

**当日验收标准**

- 给定伪造动作输入时，可以稳定生成 `/vla/goal_pose`。
- 越界输入会被裁剪或拒绝。
- 坐标系定义统一，不出现 frame 混乱。

**当日产出**

- 工作空间边界约定。
- 坐标系约定。
- 动作到目标位姿的映射规则。

### Day 8：明确执行桥职责

**今日目标**

明确 C++ `moveit_executor` 节点职责，并固定执行接口。

**核心任务**

- 规划 `moveit_executor` 节点。
- 让它只订阅 `/vla/goal_pose`。
- 内部通过 MoveIt 做规划与执行。
- 禁止直接发送裸关节命令。
- 定义失败返回与状态反馈接口。

**当日验收标准**

- 节点职责清晰。
- 失败状态清晰。
- topic 接口没有歧义。

**当日产出**

- 执行桥模块说明书。
- 状态返回约定。

### Day 9：验证目标位姿到执行的逻辑闭环

**今日目标**

验证“目标位姿 -> MoveIt 规划 -> 控制器执行”这条链路在逻辑上可闭合。

**核心任务**

- 用人工构造的 `PoseStamped` 作为输入。
- 走一遍规划与执行流程。
- 定义失败状态枚举。
- 明确拒绝条件和失败日志。

**当日验收标准**

- 可达目标能成功执行。
- 不可达目标只返回失败。
- 控制器、Gazebo 和 MoveIt 不出现异常退出。

**当日产出**

- 执行桥验收标准。
- 失败处理规则。

### Day 10：补在线视觉输入方案

**今日目标**

**当日验收标准**

- 在线图像链路方案明确。

### Day 13：设计评估与记录方法

**核心任务**

## 五、总验收条件

### 环境层

- Python VLA 环境可独立运行。
- 不破坏现有 ROS 2 工作区。

### 推理层

- 能完成“单张图片 + 文本指令 -> `ActionChunk`”。

### 映射层

- 能完成“`ActionChunk.delta_xyz` -> 安全 `goal_pose`”。

### 执行层

- 能完成“`goal_pose` -> MoveIt 规划执行 -> 控制器动作”。

### 闭环层

- 能在 Gazebo 中对单个静态目标完成低频视觉闭环接近。

## 六、重要接口与默认值

### 新增接口

- `/vla/task_text`
  - 类型：`std_msgs/msg/String`
  - 作用：发送任务文本
- `/vla/action_delta`
  - 类型：`geometry_msgs/msg/TwistStamped`
  - 作用：承载 VLA 输出动作增量
- `/vla/goal_pose`
  - 类型：`geometry_msgs/msg/PoseStamped`
  - 作用：承载安全裁剪后的目标位姿
- `/vla/exec_result`
  - 类型：`std_msgs/msg/String`
  - 作用：反馈执行状态

### 默认安全参数

- 单步位移上限：`2 cm`
- 工作空间 `x`：`[0.15, 0.70]`
- 工作空间 `y`：`[-0.35, 0.35]`
- 工作空间 `z`：`[0.20, 0.80]`
- 闭环频率：`0.5 ~ 1.0 Hz`
- 每次任务最大步数：`5`

## 七、假设与默认项

- 当前阶段只做仿真，不接真机。
- 当前阶段没有夹爪，所以只做“接近目标”。
- 默认只用单 RGB 相机，不用深度相机。
- 默认优先用 Octo 做第一阶段轻量验证。
- `Plan.md` 只记录实施计划，不记录每日执行日志。
- 每日执行中遇到的实际问题继续记录到 `Note.md`。

## 八、超期处理策略

- 如果 Day 4 前卡住，优先排环境，不提前接 ROS。
- 如果 Day 7 前卡住，优先验证动作映射，不提前接 MoveIt 执行。
- 如果 Day 10 后卡住，允许先做离线图片伪闭环，不强求当天补完在线相机。
- 如果 Day 15 结束仍未闭环成功，最小保底成果应为：
  - 离线单图推理可用
  - ROS 动作发布可用
  - 可达位姿执行链路可用

## 九、执行提醒

请始终按下面顺序推进，不要反过来：

1. 先保住现有仿真和 MoveIt 基线。
2. 再打通离线单图推理。
3. 再接 ROS 动作发布。
4. 再接目标位姿映射。
5. 最后再做低频闭环。

这条顺序的意义是：先把“眼睛”和“大脑”调通，再把“指令”翻译成机械臂能理解的目标，最后才让整套系统真正闭环。这样出问题时，排查范围会非常清楚，不会把视觉、模型、ROS、MoveIt、控制器五类问题混在一起。
